Synopsis

  We want to layer an application framework on top of the DAQRocNode class, which supports application specific response to a wide variety of external events. Any required synchronization shall be performed in a manner transparent to the application specific code.

Event List

  The framework will be able to respond to the following sorts of events:

  • Location monitors – spins on changes to a volotile memory location.
  • Fd event – File descriptor events.
  • Spectrodaq buffer arrival
  • Spectrodaq out of band buffer arrival (alarms).
  • Timer events
  • Xt event loop dispatch
  • Tcl/Tk event loop dispatch

Common Requirements

Synchronization is Transparent

Any asynchronism inherent in this structure must be hidden completely from the handlers of these events which is software written by physicists.

 

Common Features of Events

Event objects have unique names.

   Events can be:

 

Established  - which involves making them known to the system and active.

Disabled      - temporarily disables response to an event.

Enabled       - Enables response to an event that was disabled.

Destroyed   - which deactivates the event and removes it from the system.

Queried      - which produces a textual string describing in 'English' the type/state of the object

Event registry

  An event registry will be produced containing all established events.  This allows programs to determine the set of events that are being managed.  Te event registry will prevent the definition of two events with identical names.  To attempt such a duplicate registry will throw an exception.

 

Application Specific processing

Associated with an event is a function like object called an "Event Processor".  This provides the application specific code that executes when the event is fired.  For more requirements see the section describing event processors.

 

Requirements for Location Monitors

    Location monitors provide a low latency mechanism to monitor changes in volatile memory locations.  Location monitors can be set on any legitimate process virtual memory

  Location monitors have associated with them a predicate object which determines when the location monitor’s event fires.  The predicate is an application specific class/object, however the following set of predicate objects shall be provided:

  • location<type> changed since last observed.
  • (location<type> & mask<type>) == value<type>
  • location<type> == value<type> (same as above when all bits are set in mask<type>

  The predicate associated with a location monitor can be changed at any time.

Requirements for Predicates:

  • Predicates have unique names
  • Predicates implement a function call operator of the following sort:
    bool operator()(void* location)
    this operator returns TRUE if the location monitor should fire its event.

 

Requirements for Fd events

Fd events allow events to fire for changes of state on file descriptors.  Fd events fire on any of the following conditions:

  • Fd Readable without blocking.
  • Fd Writeable without blocking
  • Fd has exception condition.

These conditions closely match that of the select(2) service.

Fd event monitors shall include the capability of a timed wait or poll.

 

Requirements for Timer Events

Timer events allow software to perform single shot timed events or repeated timed events.  Timer resolution is specified in ms. Timer events have the following features:

·        Timers are specified to fire at times relative to now.

·        Single shot events may be requested, after the event fires once, it becomes disabled.

o       A Single shot event may be re-enabled with the same or different time period after it fires.

·        Multishot events may be requested, after the event fires it resets itself.

 

Requirements for Spectrodaq Buffer Events

Spectrodaq buffer arrival provides a mechanism to interface applications with the Spectrodaq server.   The Spectrodaq buffer arrival event:

  • Manages access to a single inbound Spectrodaq buffer.
  • Allows the programmer to configure the link manager to receive arbitrary sets of buffers into this buffer from an arbitrary set of links.
  • Fires each time a buffer has been received.

 

Requirements for Spectrodaq Out of Band Buffer events

The requirements for Spectrodaq Out of Band Buffer events (Alarms) is the same as for Spectrodaq Buffer events, however only out of band buffer events are selected.

Requirements for Xt event loop dispatch

The application may create exactly one Xt event loop.  The event loop is synchronized prior to dispatching an Xt event. as all other events.  This requirement allows programs to implement a GUI based on X11 widget toolkits that are built on top of Xt.

·        Creation of an Xt event loop forbids the creation of a TkInteractive interpreter since Tk Interpreters are based on X11 in the Unix environment.

Tcl/Tk event loop dispatch

The application may create exactly one Tcl or Tk interactive interpreter (See TCL facilities). 

·        Creation of a Tcl interactive interpreter forbids the creation of a Tk interactive interpreter and vica-versa

·        Creation of a Tk interactive interpreter forbids the creation of an Xt event loop and vicaversa, since Tk is X-11 based in the unix environment.

·        Synchronization of Tck/Tk interactive interpreters is prior to execution of extension commands.

o       Extension commands are implemented by deriving from DAQTCLProcessor

o       DAQTCLProcessor synchronizes prior to dispatch to its command processor.

 

Event Processors

Event processors have the following characteristics:

  • Event processors have unique names
  • An event processor may be registered on more than one compatible event
  • An event processor is able to provide the set of events on which it has been registered
  • It will be possible to programmatically inquire the set of event processors in existence.

 

 

 

Additional Support Functionality

 

Tcp/Ip client server support

 

TCP Client support

Simplified TCP/IP client support shall be provided:

  • TCP Connections to a server can be created by providing a hostname and port name or number.
  • TCP Options socket options can be individually set.
  • The socket fd can be retrieved for use within an Fd event.

TCP Server Support

TCP server support includes:

·        Creation of a service listening on a particular port.

·        An event which fires an event handler when connections are received.

·        Support for Predicates to validate the acceptance of a connection (connection security models).

·        Pre-implemented connection acceptance predicates include:

o       Unconditional accept

o       Host access list

o       Local Username/Password

·        Support for the creation of a server instance event which, for a connection, monitors the connection fd for readability and fires an event on readability.

 

TCL/Tk support:

The following TCL/Tk support will be provided:

·        Support for a “Free standing” TCL interpreter

·        Support for a TCL interpreter connected to stdin/stdout

·        Support for a TK interpreter connected to stdin/stdout and the DISPLAY environment variable.

Free Standing interpreters:

Freestanding interpreters are interpreters that must be fed scripts via method calls.

·        Interpreters have names

·        Interpreter names are unique

·        The set of interpreters can be queried

·        Two forms of script synchronization shall be supported:

o       Entire script is synchronized.

o       Synchronization occurs only when DAQTCLProcessor derived command extensions execute.

Interactive Interpreters

Interactive Interpreters supply a full interactive Tcl/Tk based command/control language for the application. 

·        Support shall be provided for an extended core Tcl interactive interpreter

·        Support shall be provided for an extended core Tk interactive interpreter

·        The instantiation of Tcl, Tk, interactive interpreters and Xt event loops are all mutually exclusive.

·        Synchronization of interactive interpreters is at the time of execution of a DAQTCLProcessor derived command extension.

·        Support for synchronized script execution shall be provided by the “built in” synchronized Tcl command extension.  This command shall have the form:
   synchronized script
the entire script is executed synchronized to the application.

Characteristics of Syncrhonization

As it seems clear that the implementation of this event framework will need to be multithreaded, it will be necessary to provide a synchronization model for this framework.  The model shall have the following features:

·        Synchronization shall be transparent to “Physicist written code”

·        Synchronization shall be recursive
